## Overview

Afenda RBAC uses **registered permission keys** from the `@afenda/permissions` package registry. Roles bundle keys; memberships attach roles to users within tenant scope. The catalogs on this page are **build-time synced** — they reflect what the ERP enforces at runtime.

<Callout type="warn" title="Least privilege">

Start with read-only keys. Add manage, post, or approve keys only when a documented workflow requires them. Review assignments after org changes (reorgs, contractor offboarding).

</Callout>

## Configure roles

<Steps>

<Step>

### Open Roles

Open **System Admin → Roles** (`/system-admin/roles`).

</Step>

<Step>

### Attach minimum keys

Create or edit a role and attach the minimum permission keys required for the job function.

</Step>

<Step>

### Prefer read-only first

Prefer read-only keys (`*_read`) until a workflow requires manage/post/approve actions.

</Step>

<Step>

### Document elevated access

Document why elevated keys exist — auditors and future admins need context.

</Step>

</Steps>

## Audit permission coverage

<Steps>

<Step>

### Open Permissions

Open **System Admin → Permissions** (`/system-admin/permissions`).

</Step>

<Step>

### Compare assignments

Compare role assignments against the governed key catalog below.

</Step>

<Step>

### Remove unused keys

Remove unused keys from roles to reduce privilege surface.

</Step>

<Step>

### Re-audit after upgrades

Repeat after module upgrades — new keys may appear in the synced catalog.

</Step>

</Steps>
