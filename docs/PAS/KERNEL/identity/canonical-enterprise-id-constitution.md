## Purpose

This document is the architecture companion to ADR-0021. It explains how Afenda governs **cross-package enterprise identifiers** without duplicating PAS normative rules.

Kernel owns the **constitution**. Domain packages own master data lifecycle. Database owns uuid persistence per [postgres-split-id-model.md](./postgres-split-id-model.md).

## Format

```
<prefix>_<ulid_body>
```

- **Prefix:** exactly three lowercase ASCII letters from `ID_FAMILIES` (`[a-z]{3}`; examples: `emp`, `cus`, `ten`)
- **Body:** 26-character Crockford base32 ULID (`[0-9A-HJKMNP-TV-Z]{26}`)

**Minting (runtime):** Kernel defines `CanonicalIdBodyGenerator` and `createCanonicalId(family, generator)` but does not ship production CSPRNG. Production ULID bodies are generated by `@afenda/database` (`generateUlidBody`) and wired at the **`apps/erp` composition root** per [ADR-0024](../adr/ADR-0024-canonical-id-body-generation.md). Tests use `createFixtureCanonicalIdBodyGenerator` or package-local canonical fixtures.

The production ULID body generator must be zero-dependency, must produce sortable 26-character Crockford base32 bodies, must use cryptographically strong randomness where available, and must not use `Math.random()`.

Every family requires:

| Capability | Module (target) |
|------------|-----------------|
| Type | `identity/families/*-id.contract.ts` |
| Registry row | `identity/registry/id-family.registry.ts` |
| Parser | `identity/canonical/canonical-id-parser.contract.ts` + family exports |
| Validator | `identity/canonical/canonical-id-validator.contract.ts` |
| Generator (if `generates: true`) | `identity/canonical/canonical-id-generator.contract.ts` + injected `CanonicalIdBodyGenerator` ([ADR-0024](../../adr/ADR-0024-canonical-id-body-generation.md)) |
| Wire ingress | `identity/wire/identity-wire.contract.ts` |

## Registry rules

1. No unregistered ID family on the platform floor.
2. No duplicate prefix across families.
3. Every registry row distinguishes `owner` from `recordOwner`.
   - `owner` identifies the architectural authority for the ID contract.
   - `recordOwner` identifies the package/domain that owns lifecycle semantics for records using that ID family.
4. Every `generates: true` family has parser + generator + tests.
5. Every parser family has a registry row and wire normalizer.
6. Fiscal IDs remain off platform floor until ADR promotion.

Full promotion workflow: [identity-promotion-process.md](./identity-promotion-process.md).

## Trust boundaries

| Boundary | Rule |
|----------|------|
| API route / query | `parseEmployeeId(wire)` — throws on invalid |
| ERP / domain code | Import from `@afenda/kernel` — never local `type X = string` |
| Database write | Generate via `create*` at governed path; store in `enterprise_id text` |
| RLS | uuid `tenant_id` only — see postgres split-ID doc |

Family parsers are for trusted ingress normalization and must throw on invalid canonical IDs. Non-throwing validators must exist for UI validation, diagnostics, governance checks, and optional defensive checks.

---

## Forbidden patterns

- `as CustomerId` outside Kernel test fixtures
- Local `type CustomerId = string` in consumer packages
- Routing `MYR` or `Asia/Kuala_Lumpur` through enterprise ID parser
- `FiscalCalendarId` / `FiscalPeriodId` on Kernel platform floor — canonical home KV-ACCT only ([ADR-0031](../../adr/ADR-0031-fiscal-domain-id-authority.md))
- Duplicate `platform-id*.ts` registries after Slice B migration
- `Math.random()` in canonical enterprise ID generation
- Drizzle schema or migration logic inside `@afenda/kernel`
- Using throwing parsers for ordinary UI form validation where non-throwing validators are required
